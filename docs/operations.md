## What lives in `.env`

`.env` is intentionally tiny. The install script seeds the secrets — you only ever edit it by hand to override a port.

| Variable | What it does | Notes |
|---|---|---|
| `SECRET_KEY` | JWT signing + cookie integrity | **Must** be 32+ bytes of true randomness. Generated by `./install.sh`; backend refuses to boot with the dev placeholder. |
| `POSTGRES_PASSWORD` | Postgres superuser password | Generated by `./install.sh`. Only ever read by the backend container over the internal Docker network. |
| `SEARXNG_SECRET` | SearXNG instance secret | Generated by `./install.sh`. |
| `NGINX_HTTP_PORT` / `NGINX_HTTPS_PORT` | Host ports nginx binds to | Default `8087` / `8488`. Override if those clash. |
| `OLLAMA_HOST_PORT` | Host port the bundled Ollama publishes | Default `11435`. |
| `COMPOSE_PROFILES` | Set to `gpu` to swap the bundled CPU Ollama for the NVIDIA variant | Off by default. |
| `SINGLE_USER_MODE` | Bypass auth entirely for solo personal setups | **Leave `false`** for any multi-user deployment. When `true` the backend stops checking JWTs. |

Everything else — public domain, model providers, search providers, push notifications, SMTP, MFA, per-user quotas — is configured in the first-run wizard or under **Admin → Settings**. The CORS allow-list is dynamic: the backend reads `app_settings.public_origins` from the DB on every request, so adding a new domain takes effect on the very next preflight without a restart.

## Unraid install

The base `docker-compose.yml` is already Unraid-friendly — uncommon ports (`8087` / `8488`) so it doesn't collide with the Unraid web UI or other Community Apps containers, and bind-mounted data under `./data/` so the CA Appdata Backup plugin and Krusader can see everything directly.

### 1. Install the Compose Manager plugin

**Apps → search "Compose Manager" → Install**. (Author: dcflachs / dlandon, via Community Apps.) Restart the UI if prompted.

### 2. Pull the project into appdata

SSH into the Unraid box (or open **Tools → Terminal**):

```bash
mkdir -p /mnt/user/appdata/promptly
cd /mnt/user/appdata/promptly
git clone https://github.com/tristenlammi/Promptly.git .
```

(Trailing `.` clones the repo *contents* into the existing folder.)

### 3. Run the install script

```bash
./install.sh
```

Seeds `.env` with strong secrets, builds the images, and starts the stack. Exits green when the backend health check passes, or prints the last 60 lines of logs and a suggested fix if something's wrong.

### 4. Register the stack in Compose Manager (optional)

**Docker → Compose → Add New Stack**

- **Stack name:** `Promptly`
- **Stack location:** `/mnt/user/appdata/promptly`
- **Compose file(s):** `docker-compose.yml`

Save. The stack is already running from step 3 — this just hooks it into the plugin so you can stop / start / view logs from the Unraid UI.

### 5. Open it

- `http://tower.local:8087` — Promptly (or your Unraid IP, e.g. `http://192.168.50.247:8087`)
- `http://tower.local:8488` — HTTPS (only useful once a cert is wired into `nginx/`)

The other containers (Frontend, Backend, Postgres, Redis, SearXNG, Collab, Ollama) stay on the internal `promptly` Docker network with no host ports — your database is never exposed on the LAN.

### Where your data lives

```
/mnt/user/appdata/promptly/
├── data/
│   ├── postgres/      # database (back this up)
│   ├── redis/         # cache + rate-limit counters
│   └── uploads/       # user files + AI-generated artefacts (back this up)
├── searxng/           # search engine config (in git)
└── docker-compose.yml
```

The CA Appdata Backup plugin automatically picks up `/mnt/user/appdata/promptly/` — configure it to **stop containers before backup** so Postgres flushes cleanly.

### Optional — front with HTTPS

Easiest: pair with the **SWAG** container (Community Apps → SWAG by linuxserver.io) on the same `promptly` network, then drop into `swag/nginx/proxy-confs/promptly.subdomain.conf`:

```nginx
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    server_name chat.*;
    include /config/nginx/ssl.conf;
    client_max_body_size 0;

    location / {
        include /config/nginx/proxy.conf;
        resolver 127.0.0.11 valid=30s;
        set $upstream_app Promptly-Nginx;
        set $upstream_port 80;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;
    }
}
```

Cloudflare Tunnel works equally well — run the `cloudflared` container, point it at `http://Promptly-Nginx:80`, let Cloudflare terminate TLS at their edge. Then paste the public URL into the wizard's public-URL step (or **Admin → Settings**).

### Day-2 on Unraid

| Task | How |
|---|---|
| **Update** | `cd /mnt/user/appdata/promptly && git pull && docker compose build && docker compose up -d` |
| **Backup** | CA Appdata Backup plugin nightly. Set "stop containers before backup". |
| **Move appdata to a different share** | Stop the stack, `mv /mnt/user/appdata/promptly /mnt/user/<new-share>/promptly`, point the Compose Manager stack location at the new path, restart. |
| **View logs** | `docker logs -f Promptly-Backend` (or the Compose Manager log window) |
| **Reset everything (wipe DB)** | `docker compose down && rm -rf data/` |

## TLS termination

The shipped `nginx/nginx.conf` listens on port 80 only. Pick one of:

1. **Cloudflare Tunnel (recommended for small deployments).** Install `cloudflared`, run `cloudflared tunnel --url http://localhost:8087` against your domain. Cloudflare terminates TLS at their edge and tunnels to nginx on plain HTTP over the loopback. No certs to renew, free for personal use.
2. **Caddy / Traefik in front of nginx.** Run them on the host with `:443` and proxy to `localhost:${NGINX_HTTP_PORT}`. They handle Let's Encrypt automatically.
3. **Add `:443` to nginx directly.** Drop a fullchain + key into `nginx/ssl/`, add a `server { listen 443 ssl; ... }` block to `nginx/nginx.conf`, and run `certbot` on the host. More moving parts; only worth it if you can't use #1 or #2.

## Manual install (without the script)

```bash
cp .env.example .env
# Generate three secrets and paste them into the matching lines:
openssl rand -hex 32   # SECRET_KEY
openssl rand -hex 32   # SEARXNG_SECRET
openssl rand -hex 24   # POSTGRES_PASSWORD
docker compose up -d
```

Alembic migrations run automatically on backend boot via `backend/entrypoint.sh`, so there's no separate "init the database" step.

## Day-2 operations

### Health

`GET /api/health` (proxied through nginx) returns `200` only when Postgres, Redis, and SearXNG are all reachable. JSON body breaks down per-component status. Wire this into your uptime monitor (UptimeRobot, BetterStack, Healthchecks.io).

```bash
curl -s https://chat.example.com/api/health | jq .
```

The compose file also runs container-level healthchecks. `docker compose ps` shows `(healthy)` against each service when things are good.

### Backups

Two volumes hold all stateful data:

- `./data/postgres/` — every conversation, message, account, audit row, settings.
- `./data/uploads/` — every uploaded and AI-generated file (images, PDFs).

A nightly cron is enough for a small deployment:

```bash
# /etc/cron.daily/promptly-backup
#!/bin/sh
set -eu
BACKUP_DIR=/var/backups/promptly
mkdir -p "$BACKUP_DIR"
DATE=$(date +%Y%m%d-%H%M%S)

# Postgres logical dump (compressed, restorable across PG versions).
docker compose -f /srv/promptly/docker-compose.yml exec -T postgres \
  pg_dump -U promptly --format=custom promptly \
  | gzip > "$BACKUP_DIR/db-$DATE.sql.gz"

# Uploaded + generated files.
tar czf "$BACKUP_DIR/uploads-$DATE.tar.gz" -C /srv/promptly/data uploads

# Keep 14 days.
find "$BACKUP_DIR" -name 'db-*.sql.gz'      -mtime +14 -delete
find "$BACKUP_DIR" -name 'uploads-*.tar.gz' -mtime +14 -delete
```

### Restore

```bash
# Database
gunzip -c db-YYYYMMDD-HHMMSS.sql.gz \
  | docker compose exec -T postgres pg_restore -U promptly -d promptly --clean --if-exists

# Uploads
tar xzf uploads-YYYYMMDD-HHMMSS.tar.gz -C /srv/promptly/data
```

### Upgrade

```bash
git pull
docker compose build
docker compose up -d
```

Alembic migrations run on boot. If you're pulling across a major release, read the release notes first — breaking schema changes should call it out.

### Logs

JSON-structured, one line per event. Every event has `ts`, `level`, `message`, and correlation IDs (`request_id`, `user_id` when authenticated).

```bash
docker compose logs backend --since=1h --no-color \
  | jq -r 'select(.level=="ERROR") | "\(.ts) \(.user_id // "-") \(.route // "-") \(.message)"'
```

Admins also get an in-app **Live Console** (Admin → Console) with the same data + filtering + error grouping.

### Common operational tasks

```bash
# Open a Postgres shell
docker compose exec postgres psql -U promptly -d promptly

# Open a Python shell with the backend env
docker compose exec backend python

# Force-rebuild just one service after a code change
docker compose build backend && docker compose up -d backend

# Run an ad-hoc Alembic command (e.g. preview the next migration)
docker compose exec backend alembic current
docker compose exec backend alembic history --verbose
```

## What's in the repo

- **`backend/`** — FastAPI, SQLAlchemy 2.0 async, Alembic. Schema migrations live in `backend/alembic/versions/` and run on container boot.
- **`frontend/`** — React 18 + Vite + TypeScript + TanStack Query + Zustand + Tailwind. Built into a static bundle and served by the `frontend` nginx.
- **`collab/`** — Hocuspocus WebSocket server for the collaborative document editor. Persists Y.js CRDT state to Postgres.
- **`nginx/`** — Public-facing reverse proxy. Routes `/api/*` to the backend, `/api/collab/*` to the collab server, everything else to the frontend.
- **`searxng/`** — Self-hosted SearXNG metasearch instance for the web-search tool. No tracking, no Google Programmable Search dependency by default.
- **`docker-compose.yml`** — Production compose. Backend, frontend, collab, postgres, redis and searxng are not exposed on host ports; only nginx is.

## Troubleshooting

**Backend refuses to start with "Refusing to start with insecure configuration".** The `SECRET_KEY` in `.env` is still the dev placeholder or under 32 chars. Re-run `./install.sh` and it'll seed a strong value.

**`SearXNG` container unhealthy on first boot.** SearXNG sometimes can't write `/etc/searxng/uwsgi.ini` under `cap_drop: ALL`. Workaround: temporarily comment out the `cap_drop` block in `docker-compose.yml`, `docker compose up -d searxng`, then restore the cap-drop and recreate.

**Login fails immediately with "Too many requests".** The Redis-backed rate limiter is intentionally aggressive. Wait 60 seconds, or in dev flush counters with `docker compose exec redis redis-cli FLUSHDB`.

**Push notifications silently missing.** The bundled VAPID keypair is auto-generated on first boot and stored in the DB. If push ever stops working, check `Admin → Settings` shows a VAPID public key — if blank, restart the backend (`docker compose restart backend`) and the bootstrap step will regenerate.

**Upgrades break open browser tabs with stale React errors.** The service worker takes control of open tabs immediately on activation, so a hard refresh (Ctrl+Shift+R) usually fixes it. If it persists, `docker compose restart frontend`.

**"Network Error" toast in Study Mode on the production URL.** Almost always a reverse-proxy HTTP/3 (QUIC) issue rather than anything in Promptly. Disable HTTP/3 on the proxy (Cloudflare dashboard → *Network* → HTTP/3 off) and retry.
